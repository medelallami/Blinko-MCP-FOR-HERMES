# Secure Credential Storage for Blinko MCP

## Best Practices for Hiding Credentials

### 1. Never Store Tokens in Config Files
- Hermes redacts Authorization headers in configuration files for security
- Storing tokens directly in config.yaml will result in `***` values
- This prevents accidental exposure in logs, backups, or version control

### 2. Use Hermes Credential Store (Recommended)
- Store tokens in `~/.hermes/.env` file
- Set restrictive file permissions: `chmod 600 ~/.hermes/.env`
- Example entry:
  ```
  BLINKO_TOKEN=<YOUR_TOKEN>
  ```

### 3. Environment Variables
- Export as environment variable: `export BLINKO_TOKEN=<YOUR_TOKEN>`
- Or configure in your deployment environment
- Ensure variables are not logged or exposed in process listings

### 4. MCP Client Usage Requirements
When using the Blinko MCP server, credentials must be provided in TWO places:
1. **Authorization Header**: `Authorization: Bearer <YOUR_TOKEN>`
2. **Tool Arguments**: Include `token` parameter in every tool call

### 5. Example Secure Workflow
```yaml
# ~/.hermes/config.yaml (MINIMAL - URL only)
mcpServers:
  blinko:
    url: http://<YOUR_SERVER_IP>:<PORT>/mcp
```

```bash
# ~/.hermes/.env (SECURE - 600 permissions)
BLINKO_TOKEN=<YOUR_TOKEN>
```

### 6. Verification
- Confirm tokens are NOT visible in Hermes config
- Check that `.env` file has proper permissions
- Test MCP connection to verify credentials work
- Monitor logs to ensure no accidental token leakage
